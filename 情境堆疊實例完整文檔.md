## 基礎情境（單一事件）

### 情境 A：會員新增更早預約

**背景資料**：
```
RecordNo: BC002183
CompanyId: 板橋光澤醫美
SYSGroup: 1
現有最早預約 A1: 2025-12-05(五) 14:30
  └─ SysSend: 2025-12-04(四) 18:00
  └─ SysSendStatus: 1 (排程中)
```

**觸發條件**：
- 會員新增比 A1 更早的預約時間：**2025-12-05(五) 10:30**

**系統行為**：
1. ✅ 新預約（10:30）成為最早預約
2. 🔄 自動排程簡訊：2025-12-04(四) 18:00
3. ❌ 自動取消原最早預約（14:30）的簡訊

**提示詞**：
```
您新增的預約（10:30）將成為最早預約
{確認發送簡訊}後將會自動取消以下簡訊寄送：
原最早預約（14:30）已有簡訊排程：
狀態：排程中
預計發送：12/04(四)18:00
```

**按鈕樣式**：
- 白色底黑色字 `{取消發送}`
- 綠色底白色字 `{確認發送}`

**結果**：
```
新預約 A0 (10:30):
  └─ SysSend: 2025-12-04(四) 18:00
  └─ SysSendStatus: 1

原預約 A1 (14:30):
  └─ SysSend: null
  └─ SysSendStatus: 0 (已取消)
```

---

### 情境 B：會員取消最早預約

**背景資料**：
```
RecordNo: BC002183
CompanyId: 板橋光澤醫美
SYSGroup: 1
最早預約 A1: 2025-12-05(五) 14:30 (已排程簡訊)
次早預約 A2: 2025-12-05(五) 16:00 (無簡訊)
  └─ SysSend: null
  └─ SysSendStatus: 0
```

**觸發條件**：
- 會員取消最早預約 A1 (14:30)

**系統行為**：
1. ❌ 取消 A1 的預約
2. 🔄 A2 (16:00) 自動補位成為最早預約
3. ✅ 為 A2 自動排程簡訊：2025-12-04(四) 18:00

**提示詞**：
```
您取消了最早預約（14:30）
次早預約（16:00）將成為新的最早預約
{確認發送簡訊}後將會為次早預約安排簡訊：
預計發送：12/04(四)18:00
```

**按鈕樣式**：
- 白色底黑色字 `{取消不發送}`
- 綠色底白色字 `{確認發送}`

**結果**：
```
A1 (14:30): [已取消]
  └─ SysSend: null
  └─ SysSendStatus: cancelled

A2 (16:00): 成為最早預約
  └─ SysSend: 2025-12-04(四) 18:00
  └─ SysSendStatus: 1 (排程中)
```

---

### 情境 C：遇到黑名單會員

**背景資料**：
```
會員類型: 黑名單（標記為不收簡訊）
操作: 新建預約、修改預約、取消最早預約後補發等
```

**適用時機**（合併多情境）：
1. 🆕 新建預約後，準備發送提醒簡訊時
2. ✏️ 修改預約（時間/門市）後，準備重新發送時
3. 🔄 取消最早預約，準備為次早預約補發時
4. ⏰ 新增更早預約，準備為新最早發送時
5. 📌 新增次早預約，準備保留或覆蓋既有最早發送時

**系統行為**：
1. 🚨 檢測到會員在黑名單
2. ⚠️ 顯示警告提示
3. 🤔 讓使用者決定是否仍要發送

**提示詞**：
```
此會員註記為不收簡訊的黑名單請確認是否{取消發送簡訊}
-----其餘情境往下增加-----
```

**按鈕樣式**：
- 🔴 紅底黃字帶特效 `{取消發送}`
- ⚪ 白底黑色字 `{確認發送}`

**特殊注意**：
- 此提示詞會**合併**到其他情境中（A/B/D）
- 優先級最高，會在原有提示詞之前顯示

---

### 情境 D：新增次早預約

**背景資料**：
```
RecordNo: BC002183
CompanyId: 板橋光澤醫美
SYSGroup: 1
最早預約 A1: 2025-12-05(五) 14:30
  └─ SysSend: 2025-12-04(四) 18:00
  └─ SysSendStatus: 1 (排程中)
```

**觸發條件**：
- 會員新增比 A1 更晚的預約 A2：**2025-12-05(五) 16:00**

**系統行為**：
1. ✅ A1 保持最早預約地位
2. ✅ A1 的簡訊排程保持不變
3. ❌ A2 不安排簡訊（僅最早預約發送）

**提示詞**：
```
您新增的是次早預約時間（16:00）
{取消發送簡訊}後將保留最早以下簡訊寄送：
原最早預約（14:30）已有簡訊排程：
狀態：排程中
預計發送：12/04(四)18:00
```

**按鈕樣式**：
- 🔴 紅底黃字帶特效 `{取消發送}`
- ⚪ 白底黑色字 `{確認發送}`

**結果**：
```
A1 (14:30): 最早預約（不變）
  └─ SysSend: 2025-12-04(四) 18:00
  └─ SysSendStatus: 1

A2 (16:00): 次早預約（新增）
  └─ SysSend: null
  └─ SysSendStatus: 0
```

## 進階情境（堆疊組合）

### 堆疊情境 1：新增更早預約 + 黑名單

**情境組合**: 情境 A + 情境 C

**背景**：
- 會員在黑名單
- 新增比現有最早預約更早的時間

**提示詞（堆疊）**：
```
⚠️ 此會員註記為不收簡訊的黑名單請確認是否{取消發送簡訊}

您新增的預約（10:30）將成為最早預約
{確認發送簡訊}後將會自動取消以下簡訊寄送：
原最早預約（14:30）已有簡訊排程：
狀態：排程中
預計發送：12/04(四)18:00
```

**按鈕樣式**：
- 🔴 紅底黃字帶特效 `{取消發送}` ← 優先級更高
- ⚪ 白底黑色字 `{確認發送}`

**使用者選擇**：
1. **選擇「取消發送」**：
   - 新預約不發送簡訊
   - 原最早預約的簡訊被取消
   - 結果：兩筆預約都不發簡訊

2. **選擇「確認發送」**：
   - 覆蓋黑名單設定
   - 新預約排程簡訊
   - 原最早預約的簡訊被取消

---

### 堆疊情境 2：次早預約補位 + 黑名單

**情境組合**: 情境 B + 情境 C

**背景**：
- 會員在黑名單
- 取消最早預約，次早預約需要補位

**提示詞（堆疊）**：
```
⚠️ 此會員註記為不收簡訊的黑名單請確認是否{取消發送簡訊}

您取消了最早預約（14:30）
次早預約（16:00）將成為新的最早預約
{確認發送簡訊}後將會為次早預約安排簡訊：
預計發送：12/04(四)18:00
```

**使用者選擇**：
1. **選擇「取消發送」**：
   - 次早預約不發送簡訊
   - 遵守黑名單規則

2. **選擇「確認發送」**：
   - 覆蓋黑名單設定
   - 為次早預約排程簡訊

---

### 堆疊情境 3：新增次早預約 + 黑名單

**情境組合**: 情境 D + 情境 C

**背景**：
- 會員在黑名單
- 新增比最早預約更晚的預約（次早）

**提示詞（堆疊）**：
```
⚠️ 此會員註記為不收簡訊的黑名單請確認是否{取消發送簡訊}

您新增的是次早預約時間（16:00）
{取消發送簡訊}後將保留最早以下簡訊寄送：
原最早預約（14:30）已有簡訊排程：
狀態：排程中
預計發送：12/04(四)18:00
```

**使用者選擇**：
1. **選擇「取消發送」**：
   - 次早預約不發送（預設行為）
   - 最早預約保持原排程

2. **選擇「確認發送」**：
   - 特殊情況：強制為次早預約也發送
   - 最早預約保持原排程
   - 結果：兩筆預約都會發送簡訊（違反常規）

---

### 堆疊情境 4：新增更早預約 + 最早預約沒發簡訊

**情境組合**: 情境 A + 最早預約簡訊狀態異常

**背景**：
```
最早預約 A1: 2025-12-05(五) 14:30
  └─ SysSendStatus: 0 (未發送)
  └─ 原因：可能是黑名單、手動取消、或發送失敗
```

**觸發條件**：
- 新增比 A1 更早的預約 A0: 10:30

**提示詞（堆疊）**：
```
⚠️ 注意：原最早預約（14:30）未發送簡訊
狀態：未發送 / 已取消 / 發送失敗

您新增的預約（10:30）將成為最早預約
請選擇簡訊發送方式：

○ 僅對新預約（10:30）發送簡訊
○ 不發送簡訊
○ 對兩筆預約都發送簡訊（特殊情況）
```

**使用者選擇**：
1. **僅對新預約發送**（推薦）：
   - A0 (10:30) 排程簡訊
   - A1 (14:30) 保持未發送

2. **不發送簡訊**：
   - A0 (10:30) 不發送
   - A1 (14:30) 保持未發送

3. **兩筆都發送**：
   - A0 (10:30) 排程簡訊
   - A1 (14:30) 也排程簡訊（違反 SYSGroup 規則）

---

### 堆疊情境 5：次早預約補位 + 原最早預約簡訊已送達

**情境組合**: 情境 B + 簡訊狀態「已送達」

**背景**：
```
最早預約 A1: 2025-12-05(五) 14:30
  └─ SysSendStatus: delivered (已送達)
  └─ 會員已收到簡訊
```

**觸發條件**：
- 會員致電取消 A1

**提示詞（堆疊）**：
```
⚠️ 重要提醒：
原最早預約（14:30）的簡訊已送達給會員
會員已收到提醒簡訊，但預約將被取消

您取消了最早預約（14:30）
次早預約（16:00）將成為新的最早預約

請選擇處理方式：
○ 為次早預約發送新簡訊
  （會員將收到第二則簡訊，提醒16:00的預約）
○ 不發送簡訊
  （假設會員透過電話已知悉）
```

**使用者選擇**：
1. **發送新簡訊**：
   - A2 (16:00) 排程簡訊
   - 會員會收到兩則簡訊（原14:30已送達，新16:00將送達）

2. **不發送**：
   - A2 (16:00) 不發送
   - 假設已透過電話溝通

---

### 堆疊情境 6：多次預約時間調整（連鎖反應）

**情境組合**: A → B → D 連續堆疊

**時間軸**：

**T1 - 初始狀態**：
```
A1: 2025-12-05(五) 14:30
  └─ SysSendStatus: 1 (排程中)
```

**T2 - 新增更早預約**（情境 A）：
```
A0: 2025-12-05(五) 10:30 [新增]
  └─ SysSendStatus: 1 (排程中)

A1: 2025-12-05(五) 14:30
  └─ SysSendStatus: 0 (已取消)
```

**T3 - 取消最早預約**（情境 B）：
```
A0: 2025-12-05(五) 10:30 [取消]
  └─ SysSendStatus: cancelled

A1: 2025-12-05(五) 14:30 [補位成為最早]
  └─ SysSendStatus: 1 (重新排程)
```

**T4 - 新增次早預約**（情境 D）：
```
A1: 2025-12-05(五) 14:30
  └─ SysSendStatus: 1 (維持)

A2: 2025-12-05(五) 16:00 [新增]
  └─ SysSendStatus: 0 (不發送)
```

**提示詞（T4 時刻）**：
```
您新增的是次早預約時間（16:00）

歷史記錄：
• 10:30 預約已取消
• 14:30 為當前最早預約（簡訊排程中）

{取消發送簡訊}後將保留最早預約（14:30）的簡訊：
狀態：排程中
預計發送：12/04(四)18:00
```

## 特殊狀態處理

### 狀態 1：已送達（delivered）

**特性**：
- ✅ 簡訊已成功送達會員手機
- 🔒 無法取消或修改
- ❌ 無操作按鈕顯示

**允許操作**：
- 無（唯讀狀態）

**懸浮效果**：
- 鼠標移入：不顯示操作按鈕
- 色塊：保持 100% 不透明

**適用情境**：
- 預約時間未過期，但簡訊已發送成功

---

### 狀態 2：已過期（expired）

**判斷邏輯**：
```javascript
當前時間 > 預約日期時間
```

**特性**：
- 🔒 所有簡訊操作被鎖定
- 🚫 無論原本簡訊狀態如何，都無法操作
- 🎨 整列顯示為灰色（opacity: 0.7）

**顯示規則**：
```
原狀態 → 過期後顯示
─────────────────────
未發送 → 已過期
排程中 → 已過期
已送達 → 已送達（保持）
已取消 → 已取消（保持）
發送失敗 → 發送失敗（保持）
```

**允許操作**：
- 無（所有按鈕隱藏）

---

### 狀態 3：排程中（scheduled）

**特性**：
- ⏰ 已安排發送時間
- ⚠️ 距離發送時間 < 30分鐘時鎖定

**允許操作**：
1. **距離發送時間 > 30 分鐘**：
   - ✅ 可以取消

2. **距離發送時間 ≤ 30 分鐘**：
   - 🔒 鎖定，無法取消
   - 提示：「距離發送時間不足30分鐘，無法取消」

**懸浮效果**：
- 顯示「取消」按鈕
- 如鎖定則按鈕呈現 disabled 狀態

---

### 狀態 4：已取消（cancelled）

**特性**：
- 📝 原本已排程但被手動取消
- 📮 簡訊未實際發送給會員
- 🔄 可以重新發送

**允許操作**：
- ✅ 重新發送（如果預約未過期）
- ❌ 重新發送（如果預約已過期）

**重新發送邏輯**：
```
1. 檢查預約是否過期
2. 如未過期，開啟簡訊發送彈窗
3. 重新計算發送時間（前一天18:00 或 前1小時）
4. 重新排程簡訊
```

---

### 狀態 5：發送失敗（failed）

**可能原因**：
- 📱 手機號碼錯誤
- 🚫 客戶手機停機
- 🌐 網路問題
- 🔌 電信業者拒絕

**特性**：
- ❌ 系統嘗試發送但失敗
- 🔄 可以重新發送
- 🔍 建議檢查會員電話號碼

**允許操作**：
- ✅ 重新發送（如果預約未過期）
- 🔧 建議同時更新會員電話

**處理建議**：
```
1. 檢查會員電話號碼是否正確
2. 確認會員手機是否停機
3. 如電話正確，點擊「重新發送」
4. 如持續失敗，改用電話聯絡
5. 更新聯絡狀況為「電話」
```

## 時間限制情境

### 時間情境 1：提前預約（標準流程）

**條件**：
- 預約時間 > 當前時間 + 1.5 天

**發送時間計算**：
```
發送時間 = 預約日期前一天 18:00
```

**範例**：
```
當前時間：2025-11-11 14:30
預約時間：2025-11-19 10:00
發送時間：2025-11-18 18:00
```

**簡訊內容**：
```
提醒您有預約:【羅東光澤】11/19(三) 10:00
03-9XX-XXXX
如須改約請聯繫您的專屬諮詢師
```

---

### 時間情境 2：當天緊急預約

**條件**：
- 預約時間 < 當前時間 + 1.5 天
- 預約時間 > 當前時間 + 1.5 小時

**發送時間計算**：
```
發送時間 = 預約時間 - 1 小時
```

**範例**：
```
當前時間：2025-11-12 11:00
預約時間：2025-11-12 15:00 (同一天)
發送時間：2025-11-12 14:00
```

**系統提示**：
```
💡 此為當天預約，系統已調整為療程前 1 小時發送
```

---

### 時間情境 3：極端緊急預約（立即發送）

**條件**：
- 預約時間 < 當前時間 + 1.5 小時

**發送時間計算**：
```
發送時間 = 立即發送（延遲幾分鐘）
```

**範例**：
```
當前時間：2025-11-12 13:40
預約時間：2025-11-12 15:00
發送時間：2025-11-12 13:42 (立即)
```

**系統提示**：
```
⚠️ 注意：預約時間緊迫，系統將立即發送簡訊
發送後幾乎無法取消，請確認會員資訊正確
```

**使用者選擇**：
```
○ 立即發送簡訊
○ 不發送（會員已電話確認）
```

---

### 時間情境 4：30分鐘鎖定規則

**條件**：
- 當前時間 > 發送時間 - 30 分鐘

**系統行為**：
```
🔒 鎖定「取消」按鈕
顯示：「距離發送時間不足30分鐘，系統已鎖定」
```

**範例**：
```
發送時間：2025-11-12 18:00
鎖定時間：2025-11-12 17:30

17:29 → 可以取消 ✅
17:30 → 無法取消 ❌
17:31 → 無法取消 ❌
```

**懸浮效果**：
- 鼠標移入仍顯示狀態
- 但「取消」按鈕變灰且無法點擊
- 顯示提示文字

## 多預約互動情境

### 多預約情境 1：同一會員、同一 SYSGroup

**規則**：
```
同一 SYSGroup 只有最早的一筆預約發送簡訊
```

**範例**：
```
RecordNo: BC002183
SYSGroup: 1

A1: 板橋光澤醫美 - 2025-12-05(五) 14:30
  └─ SysSend: 2025-12-04(四) 18:00 ✅ 發送
  └─ SysSendStatus: 1

A2: 板橋岩盤浴 - 2025-12-05(五) 16:00
  └─ SysSend: null ❌ 不發送
  └─ SysSendStatus: 0

A3: 板橋岩盤浴 - 2025-12-05(五) 18:00
  └─ SysSend: null ❌ 不發送
  └─ SysSendStatus: 0
```

**提示詞（新增 A2 時）**：
```
您新增的是次早預約時間（16:00）
此會員在 SYSGroup 1 已有最早預約（14:30）

{取消發送簡訊}後將保留最早預約的簡訊：
狀態：排程中
預計發送：12/04(四)18:00
```

**提示詞（取消 A1 時）**：
```
您取消了最早預約（14:30）
次早預約（16:00）將自動成為新的最早預約

{確認發送簡訊}後將會為次早預約安排簡訊：
預計發送：12/04(四)18:00

剩餘預約：
• 16:00 - 板橋岩盤浴 [將發送] ✅
• 18:00 - 板橋岩盤浴 [不發送] ❌
```

---

### 多預約情境 2：同一會員、不同 SYSGroup

**規則**：
```
不同 SYSGroup 各自獨立，分別發送最早預約的簡訊
```

**範例**：
```
RecordNo: BC002183

SYSGroup 1:
  A1: 板橋光澤醫美 - 2025-12-05(五) 14:30 ✅ 發送
  A2: 板橋岩盤浴 - 2025-12-05(五) 16:00 ❌ 不發送

SYSGroup 0:
  B1: 忠孝光澤 - 2025-12-05(五) 18:30 ✅ 發送
  B2: 忠孝光澤 - 2025-12-05(五) 20:00 ❌ 不發送
```

**提示詞（新增 A2 時）**：
```
您新增的是 SYSGroup 1 的次早預約時間（16:00）
此會員在其他 SYSGroup 也有預約：
• SYSGroup 0: 18:30 [最早，已排程] ✅

SYSGroup 1 最早預約（14:30）的簡訊將保留
{取消發送簡訊}
```

**會員收到的簡訊數量**：
- 2 則（每個 SYSGroup 各一則）

### 多預約情境 3：跨日期多預約

**範例**：
```
RecordNo: BC002183
SYSGroup: 1

Day 1 (2025-12-05):
  A1: 14:30 ✅ 發送（12/04 18:00）
  A2: 16:00 ❌ 不發送

  B1: 10:00 ✅ 發送（12/05 18:00）

**規則**：

- 不同日期視為獨立的預約組

**會員收到的簡訊**：

12/04 18:00 → 「提醒您有預約：12/05(五) 14:30」
12/05 18:00 → 「提醒您有預約：12/06(六) 10:00」
```

## 異常處理情境

### 異常情境 1：會員電話號碼錯誤

**觸發時機**：
- 簡訊發送時

**系統偵測**：
```
錯誤代碼: INVALID_PHONE_NUMBER
原因: 電話號碼格式錯誤或不存在
```

**系統行為**：
1. ❌ 簡訊發送失敗
2. 🔄 狀態更新為「發送失敗」
3. 📝 記錄錯誤原因

**提示詞**：
```
⚠️ 簡訊發送失敗
原因：電話號碼錯誤（0916-795-179）

建議處理方式：
1. 檢查並更新會員電話號碼
2. 確認後點擊「重新發送」
3. 或改用電話直接聯絡會員
```

**處理流程**：
```
1. 店經理檢查會員資料
2. 更新正確的電話號碼
3. 鼠標移到簡訊管理，點擊「重新發送」
4. 系統重新排程
5. 在新的發送時間自動發送
```

---

### 異常情境 2：網路連線失敗

**觸發時機**：
- 簡訊發送時網路中斷

**系統偵測**：
```
錯誤代碼: NETWORK_ERROR
原因: 無法連接到簡訊發送服務
```

**系統行為**：
1. 🔄 自動重試 3 次（間隔 5 分鐘）
2. ❌ 3 次失敗後標記為「發送失敗」
3. 📧 發送通知給系統管理員

**提示詞**：
```
⚠️ 簡訊發送失敗
原因：網路連線問題

系統已自動重試 3 次，建議：
1. 檢查網路連線
2. 稍後點擊「重新發送」
3. 如持續失敗，請聯繫技術支援
```

---

### 異常情境 3：會員手機停機

**觸發時機**：
- 簡訊發送時

**系統偵測**：
```
錯誤代碼: PHONE_UNAVAILABLE
原因: 會員手機停機或無法接收簡訊
```

**系統行為**：
1. ❌ 簡訊發送失敗
2. 🔄 狀態更新為「發送失敗」
3. 💡 建議改用其他聯絡方式

**提示詞**：
```
⚠️ 簡訊發送失敗
原因：會員手機無法接收（可能停機）

建議處理方式：
1. 改用電話直接聯絡會員
2. 確認會員聯絡資訊
3. 更新聯絡狀況為「電話」
```

### 異常情境 4：系統時間異常

**觸發時機**：
- 發送時間計算錯誤
- 伺服器時間不同步

**系統偵測**：
```
錯誤代碼: TIME_SYNC_ERROR
原因: 系統時間與標準時間差距過大
```

**系統行為**：
1. 🚨 暫停所有簡訊發送
2. 📢 顯示系統警告
3. 🔧 自動校正時間

**提示詞**：
```
🚨 系統時間異常警告

偵測到系統時間與標準時間不一致
為確保簡訊準時發送，系統已暫停排程

請聯繫技術人員處理：
📧 cole.cheng@dr-shine.com.tw
📞 0277097536 #77
```

### 異常情境 5：預約被刪除但簡訊已排程

**觸發時機**：
- 預約在資料庫中被直接刪除
- 但簡訊排程仍存在

**系統偵測**：
```
錯誤代碼: APPOINTMENT_NOT_FOUND
原因: 找不到對應的預約記錄
```

**系統行為**：
1. 🔍 發送前檢查預約是否存在
2. ❌ 如不存在，自動取消簡訊
3. 📝 記錄異常日誌

**處理流程**：
```
1. 系統在發送前檢查 (T - 5 分鐘)
2. 如果預約不存在：
   → 自動取消簡訊排程
   → 寫入日誌：「預約已刪除，簡訊已取消」
   → 不發送給會員
3. 如果預約存在：
   → 正常發送
```
