# 簡訊系統最終規格說明 v1.0

> **版本**: v1.0
> **更新日期**: 2025-11-20
> **目的**: 整合所有簡訊發送邏輯、客戶資格檢查規則、UI 互動模式及模擬器功能說明。

---

## 1. 核心規則

### 1.1. 資料結構
每筆預約 (`appointment`) 包含以下主要欄位：
- `id`: 唯一識別碼 (數字)
- `time`: 預約時間 (字串, HH:mm)
- `date`: 預約日期 (字串)
- `member`: 會員名稱 (字串)
- `store`: 門市名稱 (字串)
- `sysGroup`: 歸屬的 SYSGroup (數字)
- `smsStatus`: 簡訊狀態 (`none`, `scheduled`, `delivered`, `cancelled`, `failed`)
- `creationTime`: 此筆預約的建立時間 (日期物件)
- `mark`: **備註欄位 (用於關鍵字規則判斷)**

### 1.2. 客戶資格檢查 (v2.5 最終判斷流程)

系統在發送簡訊前，會依循以下 **4 個關卡**進行檢查，順序由上至下，任一關卡決定「引導取消發送」或「直接通過」，則後續流程不再進行。

#### 1.2.1. 第一關：黑名單檢查 (絕對優先)
- **規則**: 針對 `Type: 標註` 的預約，檢查 `Mark` 欄位是否包含任一黑名單關鍵字。
- **關鍵字**: `無`, `拒`, `沒`, `員工`, `醫師`, `離職`, `暫停`, `誤`, `簡訊`, `澳門`, `香港`, `大陸`, `空號`, `非本人`, `過世`, `往生`, `SPY`, `孕`, `客訴`, `歿`, `間諜`
- **結果**:
    - **如果符合** → **直接引導取消發送**，流程結束。
    - **如果不符合** → 進入下一關。

#### 1.2.2. 第二關：特殊規則檢查 (內含白名單例外)
- **規則**: 針對 `Type: 標註` 的預約，檢查 `Mark` 欄位是否符合「勿」或「不」的特殊規則。
- **匹配特殊規則條件**:
    - `(Mark 包含 '勿' AND Mark 不包含 '勿久等')`
    - **或者** `(Mark 包含 '不' AND Mark 不包含 '推'/'call'/'摳'/'電'/'聯')`
- **白名單例外檢查**:
    - **如果符合**上述任一特殊規則，**則進行白名單例外檢查**:
        - **檢查 `Mark` 是否同時包含白名單關鍵字 (`天使`, `好友`)**:
            - **是 (白名單例外)** → **直接通過**，流程結束。
            - **否** → **引導取消發送**，流程結束。
    - **如果不符合**上述特殊規則 → 進入下一關。

#### 1.2.3. 第三關：電訪銷開關檢查
- **規則**: 僅當後台的「接受電訪銷欄位開關」**打開**時，此關卡才生效。
- **檢查條件**: `promote` 欄位的值是否為 `0`, `10`, 或 `20`。
- **結果**:
    - **如果符合** → **引導取消發送**，流程結束。
    - **如果不符合** → 進入下一關。
    - **如果開關關閉** → 直接進入第四關。

<br>

| promote 值 | 狀態說明     | 開關關閉 | 開關打開 |
| :--------- | :----------- | :------: | :------: |
| 0          | 拒絕         | ✓ 發送   | ✗ 不發送 |
| 10         | 勿Call勿約   | ✓ 發送   | ✗ 不發送 |
| 20         | 勿call可line | ✓ 發送   | ✗ 不發送 |
| 其他       | NULL 或其他值  | ✓ 發送   | ✓ 發送   |

#### 1.2.4. 第四關：預設通過
- **規則**: 若預約資訊能通過前面所有關卡的檢查。
- **結果**: **正常，可發送**。

### 1.3. SYSGroup 簡訊發送規則
- **單一性原則**：在同一個 `SYSGroup` 群組內，針對 `同一天` 的所有預約，系統**只會**為時間**最早**的一筆預約排程簡訊。
- **時間衝突規則 (Tie-Breaking)**：如果多筆預約在完全相同的最早時間，則以**預約 `id` 最小**（即最先建立）的預約為絕對優先。
- **獨立性原則**：不同的 `SYSGroup` 之間完全獨立計算，互不影響 (`SYSGroup: 0` 的門市視為獨立單位)。

### 1.4. 「取消發送」按鈕規則
- **非中斷原則**：在彈出視窗中點擊 `{取消發送}` 按鈕，**不會**中斷主流程（例如新增預約）。預約依然會成功建立，只是相關的簡訊操作會被跳過。

## 2. UI 互動模式：發送預覽彈窗

所有需要使用者確認的簡訊操作，都彈出「發送預覽」視窗，清晰展示系統將要執行的動作。

### 2.1. 預覽視窗的組成元素
- **預約時間**: 預約的日期與時間。
- **預約ID**: 明確標示出此次操作針對的預約是哪一筆。
- **簡訊內容 (預覽)**: 模擬實際發送的簡訊文字。
- **預計發送時間**: 顯示此簡訊預計由系統送出的時間 (此為示意)。
- **提示文字**: 引導使用者做出決策的說明文字。
- **操作按鈕**: `{確認發送}` 與 `{取消發送}`。

### 2.2. 引導確認模式 (`guide-confirm`)
- **樣式**：`{確認發送}` 按鈕為**綠底白字**，`{取消發送}` 按鈕為白底黑字。

### 2.3. 引導取消模式 (`guide-cancel`)
- **樣式**：`{取消發送}` 按鈕為**紅底白字 (帶動畫)**，`{確認發送}` 按鈕為白底黑字。

## 3. 模擬器介面與操作邏輯

### 3.1. 介面佈局
- **步驟 1：選擇簡訊狀態**：設定初始盤面中「第一筆預約」的 `smsStatus`。按鈕本身會根據狀態顯示對應顏色。
- **步驟 2：選擇初始預約盤面**：作為「盤面產生器」，用於設定一組初始的預約狀態。
    - **單一預約 (G1)**：盤面僅有一筆 G1 的預約。
    - **兩個預約 (G1)**：盤面有兩筆 G1 的預約。
    - **多群組預約**：盤面有分屬 G1, G2 的預約。
- **操作執行 (工具箱)**：固定顯示所有可用的操作按鈕，對當前盤面執行任何操作。

### 3.2. 「操作執行」按鈕功能詳解

#### ⏰ 新增更早預約
- **行為**：新增一筆比當前最早預約更早的預約。
- **時間衝突處理**：系統使用 ID 比較邏輯自動判定最早預約。若新增時間與現有預約相同，ID 較小的預約會被視為最早。
- **彈窗**：觸發「引導確認模式」的發送預覽，顯示**新預約**的簡訊內容，並詢問是否取代舊排程。

#### ⚖️ 新增同時間預約
- **行為**：**動態尋找**當前 G1 中最早的預約時間，並在**相同時間點**新增一筆預約。
- **時間衝突處理**：這是展示時間衝突規則的核心操作。當多筆預約時間完全相同時，系統永遠優先處理 **ID 最小**（最先建立）的預約。
- **彈窗**：觸發「時間衝突」的特殊發送預覽，明確告知將優先處理 ID 較小的舊預約，並顯示**該舊預約**的簡訊預覽，詢問是否為其發送。新預約的簡訊狀態固定為 `none`。

#### 📌 新增次早預約
- **行為**：新增一筆比當前最早預約更晚的預約。
- **時間衝突處理**：系統會先使用 ID 比較邏輯正確識別真正的最早預約，即使盤面上有多筆時間相同的預約，也能準確判斷。
- **彈窗**：
    - **情況A (最早預約簡訊正常)**：觸發「引導取消模式」，提示此操作不會發送簡訊。
    - **情況B (最早預約簡訊異常)**：觸發「引導確認模式」，顯示**最早的預約**的簡訊預覽，引導使用者修正並為其發送。

#### ❌ 取消最早預約
- **行為**：**動態尋找**當前 G1 中最早的預約，並將其移除。
- **時間衝突處理**：
    - 使用 ID 比較邏輯找到要取消的最早預約（時間相同時選 ID 最小）。
    - 取消後，使用相同邏輯找到補位的次早預約。若有多筆時間相同的補位候選，自動選擇 ID 最小的。
- **彈窗**：觸發「引導確認模式」，顯示**補位預約**的簡訊預覽，詢問是否為其發送。

#### 🌐 新增跨群組預約
- **行為**：觸發**兩階段**互動。
- **時間衝突處理**：新增到目標群組後，系統會自動重新計算該群組的最早預約。若新增時間與該群組現有預約相同，將使用 ID 比較邏輯判定優先順序（ID 較小優先）。
- **第一階段**：彈窗讓使用者選擇要新增到 G1, G2, 還是 G0。
- **第二階段**：選擇後，彈出對應的「引導確認模式」發送預覽視窗，顯示新預約的簡訊內容。
