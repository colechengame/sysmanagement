🎯 核心需求

目標：避免同一天多個預約重複發送簡訊，降低成本並改善顧客體驗
⚠️ 重要：每個預約對應一封簡訊的設計不變，僅透過「跳過發送」來達到優化
✅ 前後端分工：消耗區域判斷與黑名單檢查由後端處理，前端僅需調用API並處理返回結果（true/false）

規則一：消耗區域門市群組（優先判斷）

🛡️ 簡訊發送前置檢查流程（三道關卡）

⭐ 前端核心工作：三個主要場景的判斷跳窗互動
  • 1️⃣ 新增預約 → 檢查是否需要發送簡訊
  • 2️⃣ 取消簡訊 → 檢查是否可以取消
  • 3️⃣ 重新發送 → 檢查是否可以重新發送
第一關：客戶黑名單檢查（優先）
檢查客戶的 Mark 欄位和 promote 狀態
❌ 在黑名單中 → 顯示提示並結束
✓ 通過檢查 → 繼續到第二關
第二關：預約時間優先級檢查
是否為當天在該範圍內的最早預約？
❌ 不是最早 → 跳過發送
✓ 是最早 → 繼續到第三關
第三關：用戶最終確認（前端互動）
顯示確認對話框，由用戶決定是否發送
✅ 調整原因：黑名單檢查放第一關可提早攔截不符合資格的客戶，節省後續查詢成本

🔄 次早預約補發機制

核心概念：當取消某預約的簡訊時，如果該客戶在同一天（或同一群組）還有其他預約，系統需要自動為「次早的預約」補發簡訊
⚠️ 重要原則：
  • 一個客戶在同一天（或同一群組範圍內）只能收到一封簡訊
  • 取消最早預約的簡訊時，必須檢查是否有次早預約
  • 次早預約需重新經過三道關卡檢查
  • 補發機制由後端自動觸發，前端負責顯示通知

觸發條件

// 次早預約補發的觸發條件 IF (取消的是該客戶當天最早預約的簡訊) { findNextEarliestAppointment(同一天 AND 同一範圍); IF (找到次早預約) { // 重新進行三道關卡檢查 checkBlacklist(次早預約); checkPriority(次早預約); IF (通過檢查) { autoSendSMS(次早預約); notifyUser('已為次早預約補發簡訊'); } } }

🔐 客戶資格檢查邏輯

Type: 標註 - 黑白名單規則（預設啟用）

⭐ 白名單（最高優先級）
Mark 欄位包含關鍵字：天使 或 好友 → 強制通過，可發送簡訊
💡 即使同時包含黑名單關鍵字，白名單優先
🚫 黑名單（若無白名單才生效）
Mark 欄位包含以下任一關鍵字 → 不發送簡訊
// 基本黑名單關鍵字 const BLACKLIST = [ '無', '拒', '沒', '員工', '醫師', '離職', '暫停', '誤', '簡訊', '澳門', '香港', '大陸', '空號', '非本人', '過世', '往生', 'SPY', '孕', '客訴', '歿', '間諜' ];
⚙️ 特殊處理規則
關鍵字 規則 程式邏輯
勿 包含「勿」但不包含「勿久等」→ 不發送 mark.includes('勿') && !mark.includes('勿久等')
不 包含「不」但不包含「推/call/摳/電/聯」→ 不發送 mark.includes('不') && !['推','call','摳','電','聯'].some(k => mark.includes(k))
// 完整檢查邏輯範例 function checkCustomerEligibility(customer) { const mark = customer.mark || ''; // 1. 白名單優先 if (mark.includes('天使') || mark.includes('好友')) { return { eligible: true, reason: '白名單通過' }; } // 2. 檢查基本黑名單 for (const keyword of BLACKLIST) { if (mark.includes(keyword)) { return { eligible: false, reason: `包含黑名單關鍵字: ${keyword}` }; } } // 3. 特殊規則：勿 if (mark.includes('勿') && !mark.includes('勿久等')) { return { eligible: false, reason: '包含「勿」但非「勿久等」' }; } // 4. 特殊規則：不 if (mark.includes('不')) { const exceptions = ['推', 'call', '摳', '電', '聯']; const hasException = exceptions.some(k => mark.includes(k)); if (!hasException) { return { eligible: false, reason: '包含「不」但無例外關鍵字' }; } } return { eligible: true, reason: '通過所有檢查' }; }

Type: 接受電訪/銷（前端開關控制，預設關閉）

// 開關控制邏輯 const promoteFilterEnabled = false; // 預設關閉 function checkPromoteStatus(customer) { if (!promoteFilterEnabled) { return { eligible: true, reason: 'promote過濾已關閉' }; } // 開關打開時的檢查 const blockedValues = [0, 10, 20]; if (blockedValues.includes(customer.promote)) { return { eligible: false, reason: `promote值為 ${customer.promote}，拒絕狀態` }; } return { eligible: true, reason: 'promote狀態通過' }; }
promote值 開關關閉 開關打開
0（拒絕） ✓ 發送 ✗ 不發送
10（勿Call勿約） ✓ 發送 ✗ 不發送
20（勿call可line） ✓ 發送 ✗ 不發送
其他值或NULL ✓ 發送 ✓ 發送

💻 前端實作範例（調用後端API）

⚡ 簡化設計：前端不需實作複雜判斷邏輯，只需調用後端API並處理返回結果
// 簡訊發送前檢查流程（前端實作） async function checkAndSendSMS(appointmentId, customerId) { try { // 調用後端API進行完整檢查 const checkResult = await fetch('/api/sms/check-eligibility', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ appointmentId, customerId }) }); const result = await checkResult.json(); // 處理後端返回結果 if (!result.eligible) { // 顯示錯誤提示 showErrorMessage(result.reason); return { success: false, message: result.reason, checkpoint: result.checkpoint }; } // ===== 通過檢查：顯示發送確認介面 ===== const userConfirmed = await showSMSConfirmDialog(result.smsContent); if (!userConfirmed) { return { success: false, message: '用戶取消發送' }; } // 執行發送 const sendResult = await sendSMS(appointmentId); return sendResult; } catch (error) { console.error('簡訊發送檢查失敗:', error); showErrorMessage('系統錯誤，請稍後再試'); return { success: false, message: error.message }; } } // 顯示確認對話框 async function showSMSConfirmDialog(smsContent) { return new Promise((resolve) => { // 顯示確認對話框UI const dialog = createDialog({ title: '確認發送簡訊', content: smsContent, buttons: [ { text: '確認發送', onClick: () => resolve(true) }, { text: '取消', onClick: () => resolve(false) } ] }); dialog.show(); }); } // 實際發送簡訊 async function sendSMS(appointmentId) { const response = await fetch('/api/sms/send', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ appointmentId }) }); return await response.json(); }

// 新增預約後的簡訊發送流程 async function onAppointmentCreated(appointmentId, customerId) { // 調用檢查API const result = await checkSMSEligibility(appointmentId, customerId); if (!result.eligible) { // 顯示不符合資格的提示 showWarning({ title: '無法發送簡訊', message: result.reason, type: result.checkpoint // 'blacklist' or 'priority' }); return; } // 通過檢查，顯示確認對話框 const confirmed = await showConfirmDialog({ title: '確認發送簡訊', content: result.smsContent, confirmText: '發送', cancelText: '跳過' }); if (confirmed) { await sendSMS(appointmentId); showSuccess('簡訊已排程發送'); } }

場景 2️⃣：取消簡訊（含次早預約補發機制）

// 取消已排程的簡訊（含次早預約補發處理） async function onCancelSMS(appointmentId, customerId) { try { // 1. 檢查是否可以取消（距離發送時間是否超過30分鐘） const canCancel = await checkCancelEligibility(appointmentId); if (!canCancel.allowed) { showError({ title: '無法取消簡訊', message: canCancel.reason, detail: '例如：距離發送時間不足30分鐘' }); return { success: false }; } // 2. 顯示取消確認對話框 const confirmed = await showConfirmDialog({ title: '確認取消簡訊', message: '確定要取消此簡訊發送嗎？', detail: '如果該客戶有其他預約，系統將自動為次早預約補發簡訊', type: 'warning', confirmText: '確認取消', cancelText: '保留簡訊' }); if (!confirmed) { return { success: false, message: '用戶取消操作' }; } // 3. 調用後端取消API（後端會自動處理次早預約補發） const cancelResult = await fetch('/api/sms/cancel', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ appointmentId, customerId }) }); const result = await cancelResult.json(); if (!result.success) { showError({ title: '取消失敗', message: result.message }); return result; } // 4. 顯示取消成功訊息 showSuccess('簡訊已成功取消'); // 5. 檢查並顯示次早預約補發通知 if (result.nextEarliestAppointment) { const next = result.nextEarliestAppointment; if (next.smsResent) { // 補發成功 showInfo({ title: '🔄 次早預約已補發簡訊', message: `系統已自動為 ${next.time} 的預約發送簡訊`, detail: next.reason, duration: 5000 // 顯示5秒 }); // 更新UI中的預約狀態 updateAppointmentSMSStatus(next.appointmentId, 'scheduled'); } else { // 有次早預約但未通過檢查 showWarning({ title: '次早預約未補發', message: `次早預約（${next.time}）未通過檢查`, detail: next.reason, icon: '⚠️' }); } } else { // 沒有次早預約 showInfo({ message: '該客戶當天無其他預約', duration: 3000 }); } return result; } catch (error) { console.error('取消簡訊失敗:', error); showError({ title: '系統錯誤', message: '取消簡訊時發生錯誤，請稍後再試' }); return { success: false, message: error.message }; } } // 更新預約列表中的簡訊狀態 function updateAppointmentSMSStatus(appointmentId, status) { const row = document.querySelector(`[data-appointment-id="${appointmentId}"]`); if (row) { const statusCell = row.querySelector('.sms-status'); if (statusCell) { statusCell.textContent = status === 'scheduled' ? '已排程' : status; statusCell.className = `sms-status status-${status}`; } } }
✨ 次早預約補發流程：
  1. 後端收到取消請求後，自動查詢次早預約
  2. 對次早預約執行三道關卡檢查（黑名單 → 優先級 → 自動發送）
  3. 後端返回補發結果（包含 smsResent、time、reason）
  4. 前端根據結果顯示對應通知並更新UI狀態

// 重新發送已取消或失敗的簡訊 async function onResendSMS(appointmentId, customerId) { // 重新檢查發送資格（黑名單可能已更新） const result = await checkSMSEligibility(appointmentId, customerId); if (!result.eligible) { showWarning({ title: '無法重新發送', message: result.reason, type: result.checkpoint }); return; } // 顯示重新發送確認對話框 const confirmed = await showConfirmDialog({ title: '確認重新發送簡訊', content: result.smsContent, message: '系統將重新計算發送時間並排程', confirmText: '重新發送', cancelText: '取消' }); if (confirmed) { await resendSMS(appointmentId); showSuccess('簡訊已重新排程發送'); } }

  • 區分類型：不同場景使用不同的對話框樣式（info/warning/error）
  • 操作可逆：提供明確的確認和取消選項

⚡ 前端實作重點

🎯 前端職責：
  • API調用：調用後端檢查API，傳入 appointmentId 和 customerId
  • 結果處理：根據後端返回的 eligible（true/false）決定是否繼續
  • 錯誤提示：當 eligible = false 時，顯示後端返回的 reason
  • 確認介面：通過檢查後，顯示簡訊內容確認對話框
  • 執行發送：用戶確認後，調用發送API
  • 錯誤處理：妥善處理API調用失敗的情況
🔧 後端職責：
  • 門市群組判斷：判斷預約是否為最早（消耗區域 or 同一門市）
  • 黑白名單檢查：檢查客戶的 Mark 欄位（白名單、黑名單、特殊規則）
  • Promote檢查：根據開關設定檢查 promote 狀態
  • 返回結果：返回 { eligible: boolean, reason: string, checkpoint: string }

📝 API設計重點：
  • 次早預約補發：在取消API的返回中包含補發資訊，前端無需額外請求
  • 錯誤訊息：清楚返回失敗原因（reason欄位），方便前端顯示給用戶
  • 檢查點標記：使用checkpoint標記失敗的關卡（blacklist/priority/user）
  • 狀態同步：返回最新的簡訊狀態，前端可直接更新UI

📋 Quick Reference

  • 檢查層級：3道關卡（黑名單 → 優先級 → 用戶確認）
  • 優先順序：白名單 > 黑名單 > 特殊規則 > 通過
  • 門市群組：5個群組，11個門市，優先判斷
  • 獨立門市：16個門市，採用同一門市規則
  • 次早補發：取消最早預約簡訊時，自動檢查並補發次早預約
  • 補發範圍：群組門市 → 同群組內；獨立門市 → 同門市
  • 補發檢查：次早預約需重新經過三道關卡檢查
  • 開關控制：promote過濾預設關閉
  • 前端職責：API調用 + UI互動 + 狀態更新
  • 後端職責：複雜邏輯判斷 + 次早補發處理
