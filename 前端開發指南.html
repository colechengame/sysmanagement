🚀 快速開始 - 前端工程師需要知道的事

  • ✅ 你的職責：API 調用 + UI 互動 + 狀態更新
  • ✅ 不是你的職責：複雜邏輯判斷（門市群組、黑名單規則）→ 後端處理
  • ✅ 核心目標：避免同一天重複發送簡訊給同一客戶
  • ✅ 三大操作：新增預約發送 | 取消簡訊 | 重新發送
  • ✅ 關鍵機制：取消簡訊時，次早預約自動補發（後端處理）

🛡️ 簡訊發送前置檢查流程（三道關卡）

💡 重點：前端只需調用 API，後端會自動執行這三道關卡檢查，並返回結果
第一關：黑名單檢查 checkpoint: "blacklist"
檢查客戶的 Mark 欄位和 promote 狀態
❌ 在黑名單中 → eligible: false
✓ 通過檢查 → 繼續第二關
第二關：優先級檢查 checkpoint: "priority"
是否為當天該範圍內的最早預約？
❌ 不是最早 → eligible: false
✓ 是最早 → 繼續第三關
第三關：用戶確認 checkpoint: "user"
前端顯示確認對話框，由用戶決定
✓ 用戶確認 → 發送簡訊
前端實作重點：
1. 調用 API：POST /api/sms/check-eligibility
2. 判斷 eligible：false → 顯示 reason，true → 顯示確認框
3. 用戶確認後：調用 POST /api/sms/send

📱 九大核心情境（實戰指南）

1️⃣ 新增預約 → 自動發送簡訊

觸發時機
櫃台人員新增預約後，系統自動觸發
// 前端流程 async function onAppointmentCreated(appointmentId, customerId) { // 1. 檢查發送資格 const result = await checkSMSEligibility(appointmentId, customerId); if (!result.eligible) { // 不符合資格：顯示原因 showInfo(result.reason); return; } // 2. 符合資格：顯示確認對話框 const confirmed = await showConfirmDialog({ title: '確認發送簡訊', content: result.smsContent }); if (confirmed) { await sendSMS(appointmentId); showSuccess('簡訊已排程發送'); } }
關鍵點：即使不是最早預約（eligible: false），也要顯示友善提示

2️⃣ 取消簡訊 → 次早預約自動補發

觸發時機
客戶表示不需要簡訊提醒，或發送時間錯誤
// 前端流程 async function onCancelSMS(appointmentId, customerId) { // 1. 檢查是否可取消（30分鐘規則） const canCancel = await checkCancelEligibility(appointmentId); if (!canCancel.allowed) { showError(canCancel.reason); return; } // 2. 顯示確認對話框 const confirmed = await showConfirmDialog({ title: '確認取消簡訊', message: '系統將自動檢查是否需要為次早預約補發簡訊' }); if (!confirmed) return; // 3. 調用取消API（後端會自動處理次早補發） const result = await cancelSMS(appointmentId, customerId); showSuccess('簡訊已取消'); // 4. 處理次早預約補發通知 if (result.nextEarliestAppointment?.smsResent) { showInfo(`已為 ${result.nextEarliestAppointment.time} 預約補發簡訊`); updateAppointmentSMSStatus(result.nextEarliestAppointment.appointmentId, 'scheduled'); } }
⭐ 重點：次早預約補發由後端自動處理，前端只需接收結果並顯示通知

3️⃣ 重新發送簡訊

觸發時機
之前取消的簡訊需要重新發送，或發送失敗需要重試
// 前端流程 async function onResendSMS(appointmentId, customerId) { // 重要：必須重新檢查資格（黑名單可能已更新） const result = await checkSMSEligibility(appointmentId, customerId); if (!result.eligible) { showWarning(result.reason); return; } const confirmed = await showConfirmDialog({ title: '確認重新發送', content: result.smsContent }); if (confirmed) { await resendSMS(appointmentId); showSuccess('簡訊已重新排程'); } }
關鍵點：重新發送前必須重新檢查，因為客戶狀態可能已改變

4️⃣ 修改預約時間

前端處理：偵測時間變更 → 調用 API 更新發送時間
後端處理：重新計算優先級，必要時取消並為次早預約補發

5️⃣ 黑名單客戶建立預約

API 返回：{ eligible: false, checkpoint: "blacklist", reason: "包含黑名單關鍵字: 拒" }
前端顯示：「此客戶在黑名單中，不會發送簡訊」
例外：白名單（天使/好友）優先，即使有黑名單關鍵字也會發送

6️⃣ 同一天多個預約

範例：客戶預約 10:00 臉部 + 14:00 身體
邏輯：只發 10:00 的簡訊（最早預約）
特殊情況：取消 10:00 簡訊時，自動為 14:00 補發

7️⃣ 跨群組/跨門市預約

範例 A：羅東岩盤浴 10:00 + 羅東光澤 14:00 → 同群組，只發一封
範例 B：羅東光澤 10:00 + 台中光澤 14:00 → 不同群組，各發一封
前端不需判斷：後端自動處理群組邏輯

8️⃣ 發送時間鎖定（30分鐘規則）

規則：距離發送時間 < 30 分鐘無法取消
API 返回：{ allowed: false, reason: "距離發送時間不足30分鐘", remainingMinutes: 25 }
UI 建議：取消按鈕顯示為灰色（disabled）

9️⃣ 取消預約同時取消簡訊

前端處理：取消預約確認框增加「是否同時取消簡訊」選項
邏輯：如果選擇取消簡訊，執行情境 2️⃣ 的流程

📚 補充知識（後端處理的邏輯）

💡 重點：以下內容前端不需要實作，僅供了解系統運作原理

門市群組配置

共 27 個門市：11 個群組門市（5 個群組）+ 16 個獨立門市

群組 1：忠孝岩盤浴、忠孝光澤
群組 2：羅東岩盤浴、羅東光澤
群組 3：板橋醫美、板橋岩盤浴
群組 4：中壢彤顏健保、中壢彤顏醫美、中壢岩盤浴
群組 5：台中光澤診所、台中岩盤浴

黑白名單規則

白名單（最高優先級）：Mark 包含「天使」或「好友」
黑名單關鍵字：無、拒、沒、員工、醫師、離職、暫停、誤、簡訊、澳門、香港、大陸、空號、非本人、過世、往生、SPY、孕、客訴、歿、間諜
特殊規則：「勿」但非「勿久等」、「不」但無例外字（推/call/摳/電/聯）

次早預約補發機制

觸發條件：取消最早預約的簡訊
搜尋範圍：群組門市 → 同群組內；獨立門市 → 同門市
檢查流程：次早預約需重新經過三道關卡
前端職責：接收並顯示補發結果通知

發送時間規則

標準發送：預約前一天晚上 18:00
緊急預約：療程時間前 1 小時
取消鎖定：發送時間前 30 分鐘無法取消

📋 Quick Reference

  • 🔹 三道關卡：黑名單 → 優先級 → 用戶確認
  • 🔹 三大操作：新增發送、取消簡訊、重新發送
  • 🔹 次早補發：後端自動處理，前端顯示通知
  • 🔹 30分鐘鎖定：距離發送 < 30min 無法取消
  • 🔹 白名單優先：天使/好友 > 黑名單
  • 🔹 群組門市：5 個群組，同群組只發一封
  • 🔹 獨立門市：16 個門市，同門市只發一封

  • 🔹 後端職責：複雜邏輯 + 次早補發 + 群組判斷
