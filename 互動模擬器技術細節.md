# 互動模擬器技術細節 v1.0

> **目的**: 本文件詳細說明 `完整簡訊狀態互動模擬.html` 的內部 JavaScript 實現邏輯。
> **檔案路徑**: `完整簡訊狀態互動模擬.html`

---

## 1. 全局變數與狀態管理

模擬器的核心狀態由以下全局 JavaScript 變數維護：

-   `smsStatuses`: 定義各種簡訊狀態（如 `none`, `scheduled` 等）的詳細資訊。
-   `currentStatus`: 記錄步驟 1 中選擇的「最早預約簡訊狀態」。
-   `currentScenario`: 記錄步驟 2 中選擇的「初始預約盤面」。
-   `isBlacklisted`: 布林值，表示會員是否為黑名單（透過勾選框控制）。
-   `appointments`: 核心數據陣列，包含所有當前顯示的預約物件。每個預約物件都有其 `id`, `time`, `date`, `member`, `store`, `sysGroup`, `smsStatus`, `creationTime` 等屬性。
-   `nextId`: 用於為新建立的預約分配唯一 ID。

## 2. 初始化與 UI 控制流程

### 2.1. `init()`
-   在 `DOMContentLoaded` 事件觸發時執行。
-   負責綁定所有頂部狀態/盤面選擇按鈕、黑名單勾選框和重置按鈕的事件監聽器 (`addEventListener`)。
-   呼叫 `resetState(true)` 進行初始化設置。

### 2.2. `resetState(isInitial = false)`
-   重置模擬器到預設的初始狀態。
-   取消勾選黑名單 (`isBlacklisted = false`)。
-   將步驟 1 簡訊狀態設為 `none`，步驟 2 初始盤面設為 `single-g1`。
-   呼叫 `updateDisplay()` 刷新介面。
-   在非初始化時顯示一個「重置完成」的提示 (`showToast`)。

### 2.3. `selectStatus(status, isInitial = false)`
-   更新 `currentStatus` 變數。
-   更新步驟 1 按鈕的活躍狀態 (`active` class)。
-   非初始化時觸發 `updateDisplay()`。

### 2.4. `selectScenario(scenario, isInitial = false)`
-   更新 `currentScenario` 變數。
-   更新步驟 2 按鈕的活躍狀態 (`active` class)。
-   非初始化時觸發 `updateDisplay()`。

### 2.5. `updateDisplay()`
-   每次狀態/盤面改變時呼叫，負責刷新模擬器介面。
-   從 `getInitialAppointments()` 取得初始預約數據。
-   呼叫 `recalculateEarliestAndRender()` 處理排序和渲染。
-   呼叫 `renderActionButtons()` 渲染操作工具箱按鈕。

## 3. 數據與核心邏輯

### 3.1. `getInitialAppointments(scenario, initialStatus)`
-   根據 `currentScenario` 和 `currentStatus` 返回一組預設的 `appointments` 數據。
-   此數據是「初始盤面」的定義。

### 3.2. `getEarliestInGroup(group)` - 核心時間衝突處理邏輯
-   **核心排序邏輯**，被所有操作函式使用。
-   接收一個預約陣列 (屬於同一 SYSGroup)。
-   根據預約時間 (time) 進行排序。
-   **時間衝突規則 (Tie-Breaking)**：若時間相同，則以 `id` 最小的預約（最先建立）作為次要排序條件。
-   返回該群組中唯一的最早預約物件。
-   **重要性**：此邏輯確保在所有情境中（新增更早、新增同時間、新增次早、取消最早、跨群組新增）都能一致地處理時間衝突，避免歧義。

### 3.3. `recalculateEarliestAndRender()`
-   重新計算所有預約的「最早」狀態並渲染。
-   將所有預約按 `sysGroup` 分組。
-   對每個群組，使用 `getEarliestInGroup()` 找到最早預約，並將其 `isEarliest` 標誌設為 `true`。
-   呼叫 `renderAppointments()` 更新顯示。

---

## 4. UI 渲染與互動

### 4.1. `renderAppointments()`
-   將 `appointments` 數據渲染到頁面上的預約卡片顯示區。
-   預約卡片會顯示 `id`, `time`, `date`, `member`, `store`, `sysGroup`, `smsStatus` (帶有圖標和顏色), 以及 `creationTime`。
-   使用 `apt.isEarliest` 標誌來顯示「最早預約」的徽章。
-   卡片會根據 `sysGroup` 顯示不同的邊框顏色。

### 4.2. `renderActionButtons()`
-   渲染底部「操作執行 (工具箱)」區域的所有操作按鈕。
-   每個按鈕（新增更早、新增同時間、取消最早、跨群組等）都綁定對應的 `execute...` 函式。

### 4.3. `generateSmsPreview(apt)`
-   根據傳入的預約物件 (`apt`)，生成 HTML 格式的簡訊預覽內容。
-   此內容用於 `showModal` 函式內部。

### 4.4. `showModal(body, modalType, confirmAction, cancelAction)`
-   核心彈窗顯示函式。
-   接收 `body` (包含 `title`, `text`, `preview`, `footer`)、`modalType` (決定按鈕樣式) 和確認/取消的回調函式。
-   判斷 `isBlacklisted` 狀態，若為真則強制使用 `guide-cancel` 模式。
-   動態生成彈窗 HTML，並為其中的 `{確認發送}` 和 `{取消發送}` 按鈕綁定 `confirmAction` 和 `cancelAction`。
-   處理彈窗的顯示與關閉邏輯。

---

## 5. 操作函式 (execute & confirm)

所有 `execute...` 函式負責準備數據和呼叫 `showModal()`。
所有 `confirm...` 函式則作為 `showModal()` 的回調，負責在使用者確認後執行實際的數據修改和刷新操作。

**重要**：所有操作函式都使用 `getEarliestInGroup()` 核心邏輯來尋找最早預約，確保時間衝突時的一致性處理（時間相同時優先處理 ID 較小的預約）。

### 5.1. `executeAddEarlier()` / `confirmAddEarlier(shouldSend)`
-   **功能**：處理新增比現有預約更早的邏輯。
-   **時間衝突處理**：使用 `getEarliestInGroup()` 找到舊的最早預約。若新增時間與現有預約相同，系統會自動應用 ID 比較邏輯來判定優先順序。
-   **簡訊處理**：若使用者確認發送，舊最早預約的簡訊狀態會被設為 `cancelled`，新預約設為 `scheduled`。

### 5.2. `executeAddSameTime()` / `confirmAddSameTime(shouldSend)`
-   **功能**：處理新增與現有最早預約時間相同的邏輯。
-   **時間衝突處理**：使用 `getEarliestInGroup()` 找到現有最早預約，並在彈窗中明確告知使用者將優先處理 ID 較小的舊預約。
-   **彈窗特殊說明**：顯示「時間衝突」標題，預覽舊預約（ID較小）的簡訊內容，詢問是否為舊預約修正簡訊狀態。
-   **簡訊處理**：新預約的 `smsStatus` 固定為 `none`，若使用者確認，則舊預約設為 `scheduled`。

### 5.3. `executeAddSecond()` / `confirmAddSecond(shouldSend)`
-   **功能**：處理新增次早預約的邏輯。
-   **時間衝突處理**：使用 `getEarliestInGroup()` 找到最早預約來檢查其簡訊狀態。即使新增的次早時間與其他預約相同，系統也會正確識別真正的最早預約（透過 ID 比較）。
-   **彈窗邏輯**：
    -   若最早預約簡訊異常 (`none`, `cancelled`, `failed`)：引導確認模式，詢問是否為最早預約修正。
    -   若最早預約簡訊正常：引導取消模式，提示此操作不會發送簡訊。

### 5.4. `executeCancelEarliest()` / `confirmCancelEarliest(shouldSend)`
-   **功能**：處理取消最早預約的邏輯。
-   **時間衝突處理**：
    -   使用 `getEarliestInGroup()` 找到要取消的最早預約。
    -   使用 `getEarliestInGroup()` 找到補位的次早預約。
    -   若有多筆預約時間相同，系統會自動選擇 ID 最小的作為補位預約。
-   **簡訊處理**：若使用者確認發送，補位預約的簡訊狀態會被設為 `scheduled`。

### 5.5. `executeCrossGroupAdd()` / `confirmCrossGroupAdd(group, store, shouldSend)`
-   **功能**：處理跨群組新增預約的兩階段邏輯。
-   **時間衝突處理**：新增到目標群組後，若該群組已有預約且時間相同，`recalculateEarliestAndRender()` 會自動應用 `getEarliestInGroup()` 邏輯來重新計算最早預約（時間相同時比較 ID）。
-   **階段說明**：
    -   第一階段：選擇目標群組 (G1, G2, G0)。
    -   第二階段：顯示新預約的簡訊預覽，詢問是否發送。

## 6. 工具函式

-   **`closeModal()`**: 關閉當前彈窗。
-   **`showToast(type, icon, message)`**: 顯示臨時提示訊息。
