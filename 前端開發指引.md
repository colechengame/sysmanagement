# 簡訊發送優化 - 前端開發指引

> 後端主動推送架構 | 前端專注執行 | 邏輯規則說明

---

## 🎯 核心概念

### 系統目標

避免同一客戶在同一天（同一範圍內）收到多封簡訊。

### 架構設計：後端主動推送

**傳統模式（已廢棄）：**

- ❌ 前端每次 CRUD 都要主動調用檢查 API
- ❌ 前端需要處理複雜的判斷邏輯
- ❌ API 調用次數多，效率低

**新架構（後端主動推送）：**

- ✅ 後端自動計算並推送需要處理的簡訊
- ✅ 前端只需接收推送並執行
- ✅ 前端職責極度簡化

### 前端唯一職責

```text
接收後端推送 → 顯示確認框 → 執行動作 → 回報結果
```

## 📡 後端推送事件

後端會在以下情況主動推送事件給前端：

### 事件 1：需要發送簡訊

**觸發時機：** 新增預約且該預約需要發送簡訊

**推送內容：**

```json
{
  "event": "sms_send_required",
  "appointmentId": "12345",
  "customerId": "67890",
  "smsContent": "親愛的王小姐，提醒您明天下午2點...",
  "scheduledTime": "2025-11-17 18:00:00"
}
```

**前端處理：**

1. 顯示確認對話框（顯示 `smsContent`）
2. 用戶確認 → 調用 `POST /api/sms/send`
3. 用戶取消 → 調用 `POST /api/sms/skip`

### 事件 2：簡訊已被跳過

**觸發時機：** 新增預約但該預約不需要發送簡訊（有更早預約或黑名單）

**推送內容：**

```json
{
  "event": "sms_skipped",
  "appointmentId": "12345",
  "reason": "客戶今天已有更早預約(10:00)",
  "checkpoint": "priority"
}
```

**前端處理：**

1. 顯示提示訊息（`reason`）
2. 無需任何操作

### 事件 3：次早預約需要補發

**觸發時機：** 取消最早預約的簡訊後，系統自動找到次早預約

**推送內容：**

```json
{
  "event": "resend_required",
  "appointmentId": "12346",
  "customerId": "67890",
  "smsContent": "親愛的王小姐，提醒您明天下午2點...",
  "reason": "次早預約自動補發",
  "originalAppointmentId": "12345"
}
```

**前端處理：**

1. 顯示通知：「已為 14:00 預約補發簡訊」
2. 自動調用 `POST /api/sms/send`（無需用戶確認）
3. 更新 UI 狀態

### 事件 4：簡訊取消被鎖定

**觸發時機：** 用戶嘗試取消簡訊，但距離發送時間 < 30 分鐘

**推送內容：**

```json
{
  "event": "cancel_locked",
  "appointmentId": "12345",
  "remainingMinutes": 25,
  "reason": "距離發送時間不足30分鐘"
}
```

**前端處理：**

1. 顯示錯誤訊息
2. 禁用取消按鈕

## 🔌 前端需要實作的 API

### 1. 發送簡訊（用戶確認後）

```http
POST /api/sms/send
```

**Request：**

```json
{
  "appointmentId": "12345"
}
```

**Response：**

```json
{
  "success": true,
  "message": "簡訊已排程發送"
}
```

### 2. 跳過簡訊（用戶取消發送）

```http
POST /api/sms/skip
```

**Request：**

```json
{
  "appointmentId": "12345"
}
```

**Response：**

```json
{
  "success": true,
  "message": "已跳過簡訊發送"
}
```

### 3. 取消簡訊

```http
POST /api/sms/cancel
```

**Request：**

```json
{
  "appointmentId": "12345"
}
```

**Response：**

```json
{
  "success": true,
  "message": "簡訊已取消"
}
```

**說明：** 後端會自動處理次早預約補發，並透過事件 3 推送給前端

## 💻 前端實作範例

### WebSocket 連接

```javascript
const ws = new WebSocket('ws://your-server/sms-events');

ws.onmessage = (event) => {
  const data = JSON.parse(event.data);
  handleSMSEvent(data);
};
```

### 事件處理器

```javascript
function handleSMSEvent(event) {
  switch (event.event) {
    case 'sms_send_required':
      showSendConfirmDialog(event);
      break;

    case 'sms_skipped':
      showInfoMessage(event.reason);
      break;

    case 'resend_required':
      showResendNotification(event);
      autoSendSMS(event.appointmentId);
      break;

    case 'cancel_locked':
      showErrorMessage(event.reason);
      disableCancelButton(event.appointmentId);
      break;
  }
}
```

### 發送確認對話框

```javascript
async function showSendConfirmDialog(event) {
  const confirmed = await showDialog({
    title: '確認發送簡訊',
    content: event.smsContent,
    confirmText: '發送',
    cancelText: '跳過'
  });

  if (confirmed) {
    await fetch('/api/sms/send', {
      method: 'POST',
      body: JSON.stringify({ appointmentId: event.appointmentId })
    });
  } else {
    await fetch('/api/sms/skip', {
      method: 'POST',
      body: JSON.stringify({ appointmentId: event.appointmentId })
    });
  }
}
```

### 次早預約自動補發

```javascript
async function autoSendSMS(appointmentId) {
  await fetch('/api/sms/send', {
    method: 'POST',
    body: JSON.stringify({ appointmentId })
  });

  showSuccessMessage('次早預約簡訊已自動補發');
}
```

## 📋 前端工作清單

### 必須實作

- [ ] 建立 WebSocket 連接接收後端推送
- [ ] 實作事件處理器（4 種事件）
- [ ] 實作發送確認對話框
- [ ] 實作 3 個 API 調用（send / skip / cancel）
- [ ] 實作 UI 狀態更新

### 不需要實作

- ❌ 黑名單檢查邏輯（後端處理）
- ❌ 優先級計算邏輯（後端處理）
- ❌ 門市群組判斷邏輯（後端處理）
- ❌ 次早預約查詢邏輯（後端處理）
- ❌ 主動調用檢查 API（後端推送）

## 🧪 測試案例

### 案例 1：新增預約 → 需要發送簡訊

**步驟：**

1. 櫃台新增預約
2. 後端推送 `sms_send_required` 事件
3. 前端顯示確認對話框
4. 用戶點擊「發送」
5. 調用 `POST /api/sms/send`

### 案例 2：新增預約 → 不需要發送（已有更早預約）

**步驟：**

1. 櫃台新增預約（該客戶今天已有 10:00 預約）
2. 後端推送 `sms_skipped` 事件
3. 前端顯示提示：「客戶今天已有更早預約(10:00)」

### 案例 3：取消簡訊 → 次早預約自動補發

**步驟：**

1. 用戶取消 10:00 預約的簡訊
2. 調用 `POST /api/sms/cancel`
3. 後端推送 `resend_required` 事件（針對 14:00 預約）
4. 前端自動調用 `POST /api/sms/send`
5. 顯示通知：「已為 14:00 預約補發簡訊」

### 案例 4：取消簡訊被鎖定（30分鐘規則）

**步驟：**

1. 用戶嘗試取消簡訊（距離發送時間 < 30 分鐘）
2. 後端推送 `cancel_locked` 事件
3. 前端顯示錯誤訊息
4. 禁用取消按鈕

## 🎯 關鍵邏輯規則

### 1. 後端負責所有判斷

```text
後端自動檢查：
  - 黑名單（白名單優先）
  - 優先級（是否最早預約）
  - 門市群組範圍
  - 30 分鐘鎖定規則
```

### 2. 前端只負責執行

```text
前端接收推送：
  - 顯示對話框
  - 調用 API
  - 更新 UI
```

### 3. 次早補發完全自動化

```text
取消最早預約簡訊 → 後端自動查詢次早預約 → 推送 resend_required 事件 → 前端自動發送
```

### 4. 推送事件類型

```text
sms_send_required → 需要用戶確認
sms_skipped → 僅顯示提示
resend_required → 自動執行
cancel_locked → 顯示錯誤並禁用按鈕
```
