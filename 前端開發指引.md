# 簡訊發送優化 - 前端開發邏輯指引

> 專注邏輯規則 | 前後端職責分工 | 快速參考

---

## 🎯 核心邏輯

### 系統目標
避免同一客戶在同一天（同一範圍內）收到多封簡訊，降低成本並改善體驗。

### 職責分工

**前端負責：**
- API 調用
- UI 對話框互動
- 結果處理與通知
- 狀態更新

**後端負責：**
- 門市群組判斷
- 黑名單規則檢查
- 優先級計算
- 次早預約補發

---

## 🛡️ 三道關卡檢查流程

### 第一關：黑名單檢查
- **檢查對象：** 客戶的 `Mark` 欄位和 `promote` 狀態
- **失敗返回：** `{ eligible: false, checkpoint: "blacklist", reason: "包含黑名單關鍵字: 拒" }`
- **白名單優先：** Mark 包含「天使」或「好友」→ 強制通過（即使有黑名單關鍵字）
- **黑名單關鍵字：** 無、拒、沒、員工、醫師、離職、暫停、誤、簡訊、澳門、香港、大陸、空號、非本人、過世、往生、SPY、孕、客訴、歿、間諜
- **特殊規則：**
  - 包含「勿」但不包含「勿久等」→ 不發送
  - 包含「不」但不包含「推/call/摳/電/聯」→ 不發送

### 第二關：優先級檢查
- **檢查邏輯：** 是否為當天該範圍內的最早預約？
- **範圍定義：**
  - 群組門市（5個群組）→ 同群組內所有門市
  - 獨立門市（16個門市）→ 同一門市
- **失敗返回：** `{ eligible: false, checkpoint: "priority", reason: "客戶今天已有更早預約(10:00)" }`

### 第三關：用戶確認
- **前端職責：** 顯示確認對話框，由用戶決定是否發送
- **對話框內容：** 顯示簡訊內容預覽 (`smsContent`)
- **用戶取消：** 不發送簡訊，不視為錯誤

---

## 📱 九大核心情境邏輯

### 情境 1：新增預約 → 自動發送簡訊

**觸發時機：** 櫃台人員新增預約

**邏輯流程：**
1. 調用 `POST /api/sms/check-eligibility`
2. 判斷 `eligible`：
   - `false` → 顯示 `reason`（提示訊息）
   - `true` → 顯示確認對話框
3. 用戶確認 → 調用 `POST /api/sms/send`

**關鍵點：** 即使 `eligible: false`，也要顯示友善提示（例如：「此客戶今天已有更早的預約」）

---

### 情境 2：取消簡訊 → 次早預約自動補發

**觸發時機：** 客戶表示不需要簡訊提醒

**邏輯流程：**
1. 調用 `POST /api/sms/check-cancel-eligibility` 檢查是否可取消
2. 判斷 30 分鐘規則：
   - 距離發送時間 < 30 分鐘 → `allowed: false`，顯示錯誤
   - 距離發送時間 ≥ 30 分鐘 → 可取消
3. 用戶確認 → 調用 `POST /api/sms/cancel`
4. 後端自動處理次早預約補發
5. 前端根據 `nextEarliestAppointment` 顯示通知：
   - `smsResent: true` → 顯示「已為 14:00 預約補發簡訊」
   - `smsResent: false` → 顯示「次早預約未通過檢查」
   - `null` → 顯示「該客戶當天無其他預約」

**次早預約補發邏輯（後端自動執行）：**
- 搜尋範圍：群組門市 → 同群組內；獨立門市 → 同門市
- 補發條件：次早預約需重新經過三道關卡檢查
- 前端職責：接收並顯示補發結果通知

---

### 情境 3：重新發送簡訊

**觸發時機：** 之前取消的簡訊需要重新發送

**邏輯流程：**
1. **必須重新檢查資格** → 調用 `POST /api/sms/check-eligibility`
2. 判斷 `eligible` 並顯示對應訊息
3. 用戶確認 → 調用 `POST /api/sms/resend`

**關鍵點：** 不能直接重發，必須重新檢查（客戶狀態可能已改變）

---

### 情境 4：修改預約時間

**邏輯：**
- 偵測預約時間變更
- 調用 `POST /api/sms/update-schedule` 更新發送時間
- 後端重新計算優先級，必要時取消並為次早預約補發

---

### 情境 5：黑名單客戶建立預約

**邏輯：**
- API 返回：`{ eligible: false, checkpoint: "blacklist", reason: "包含黑名單關鍵字: 拒" }`
- 前端顯示：「此客戶在黑名單中，不會發送簡訊」
- **例外：** 白名單（天使/好友）優先，即使有黑名單關鍵字也會發送

---

### 情境 6：同一天多個預約

**邏輯：**
- 客戶預約 10:00 臉部 + 14:00 身體
- 系統只為 10:00（最早預約）發送簡訊
- 14:00 預約返回：`{ eligible: false, checkpoint: "priority" }`
- **特殊情況：** 取消 10:00 簡訊時，自動為 14:00 補發

---

### 情境 7：跨群組/跨門市預約

**邏輯規則：**
- **同一群組內：** 羅東岩盤浴 10:00 + 羅東光澤 14:00 → 只發一封
- **不同群組：** 羅東光澤 10:00 + 台中光澤 14:00 → 各發一封
- **群組 + 獨立門市：** 羅東光澤 10:00 + 台北醫美 14:00 → 各發一封

**前端不需判斷群組邏輯，後端自動處理**

---

### 情境 8：發送時間鎖定（30分鐘規則）

**邏輯：**
- 距離發送時間 < 30 分鐘 → 無法取消
- API 返回：`{ allowed: false, reason: "距離發送時間不足30分鐘", remainingMinutes: 25 }`
- **UI 建議：** 取消按鈕顯示為灰色（disabled）

---

### 情境 9：取消預約同時取消簡訊

**邏輯：**
- 取消預約確認框增加「是否同時取消簡訊」選項
- 如果選擇取消簡訊 → 執行情境 2 的流程

## 🔌 API 邏輯規格

### 1. 檢查簡訊發送資格
```
POST /api/sms/check-eligibility
```

**Request：**
```json
{
  "appointmentId": "12345",
  "customerId": "67890"
}
```

**Response：**
```json
{
  "eligible": true/false,
  "reason": "通過所有檢查" | "包含黑名單關鍵字: 拒",
  "checkpoint": "blacklist" | "priority" | "user",
  "smsContent": "親愛的王小姐，提醒您明天..."
}
```

**checkpoint 說明：**
- `blacklist` - 黑名單檢查未通過
- `priority` - 優先級檢查未通過（不是最早預約）
- `user` - 需要用戶確認

---

### 2. 發送簡訊
```
POST /api/sms/send
```

**Request：**
```json
{
  "appointmentId": "12345"
}
```

**Response：**
```json
{
  "success": true,
  "message": "簡訊已排程發送",
  "scheduledTime": "2025-11-17 18:00:00"
}
```

---

### 3. 取消簡訊（含次早預約補發）
```
POST /api/sms/cancel
```

**Request：**
```json
{
  "appointmentId": "12345",
  "customerId": "67890"
}
```

**Response：**
```json
{
  "success": true,
  "message": "簡訊已取消",
  "nextEarliestAppointment": {
    "appointmentId": "12346",
    "time": "14:00",
    "smsResent": true,
    "reason": "次早預約補發成功"
  } | null
}
```

**nextEarliestAppointment 邏輯：**
- `null` - 無次早預約
- `smsResent: true` - 補發成功
- `smsResent: false` - 有次早預約但未通過檢查（例如：黑名單）

---

### 4. 重新發送簡訊
```
POST /api/sms/resend
```

**Request：**
```json
{
  "appointmentId": "12345",
  "customerId": "67890"
}
```

**Response：**
```json
{
  "success": true,
  "message": "簡訊已重新排程",
  "scheduledTime": "2025-11-17 18:00:00"
}
```

---

### 5. 檢查取消資格
```
POST /api/sms/check-cancel-eligibility
```

**Request：**
```json
{
  "appointmentId": "12345"
}
```

**Response：**
```json
{
  "allowed": true/false,
  "reason": "可以取消" | "距離發送時間不足30分鐘",
  "remainingMinutes": 45
}
```

---

### 6. 更新簡訊發送時間
```
POST /api/sms/update-schedule
```

**Request：**
```json
{
  "appointmentId": "12345",
  "newAppointmentTime": "2025-11-18 14:00:00"
}
```

**Response：**
```json
{
  "success": true,
  "newScheduledTime": "2025-11-17 18:00:00",
  "priorityChanged": false
}
```

**priorityChanged 說明：**
- `true` - 優先級改變，需要重新處理（可能取消並補發次早預約）
- `false` - 只是時間調整，不影響優先級

## 📋 關鍵邏輯規則總結

### 1. 三道關卡順序
```
黑名單檢查 → 優先級檢查 → 用戶確認
```

### 2. 白名單優先原則
```
白名單（天使/好友）> 黑名單 > 特殊規則
```

### 3. 優先級判斷範圍
```
群組門市 → 同群組內所有門市
獨立門市 → 同一門市
```

### 4. 次早預約補發條件
```
取消最早預約簡訊 → 自動查詢次早預約 → 重新三道關卡檢查 → 自動補發
```

### 5. 30 分鐘鎖定規則
```
距離發送時間 < 30 分鐘 → 無法取消
```

### 6. 發送時間計算
```
標準發送：預約前一天晚上 18:00
緊急預約：療程時間前 1 小時
```

### 7. 門市群組配置
```
群組 1：忠孝岩盤浴、忠孝光澤
群組 2：羅東岩盤浴、羅東光澤
群組 3：板橋醫美、板橋岩盤浴
群組 4：中壢彤顏健保、中壢彤顏醫美、中壢岩盤浴
群組 5：台中光澤診所、台中岩盤浴
其他 16 個獨立門市
```

### 8. 前端必須重新檢查的情境
```
重新發送簡訊 → 必須調用 check-eligibility（黑名單可能已更新）
修改預約時間 → 必須檢查優先級是否改變
```

### 9. 次早預約補發通知處理
```
smsResent: true → 顯示成功通知 + 更新 UI 狀態
smsResent: false → 顯示警告（未通過檢查）
null → 顯示資訊（無次早預約）
```

## 💡 前端實作要點

### 1. API 調用時機
- 新增預約 → 自動調用 `check-eligibility`
- 取消簡訊 → 先調用 `check-cancel-eligibility`，再調用 `cancel`
- 重新發送 → 先調用 `check-eligibility`，再調用 `resend`
- 修改時間 → 調用 `update-schedule`

### 2. 錯誤處理
- `eligible: false` 不是錯誤，是正常流程
- 應顯示友善提示，不應顯示錯誤訊息
- 真正的錯誤：API 調用失敗、網路錯誤

### 3. UI 狀態更新
- 次早預約補發成功 → 更新該預約的簡訊狀態為「已排程」
- 取消簡訊 → 更新該預約的簡訊狀態為「已取消」
- 30 分鐘內 → 取消按鈕應顯示為 disabled

### 4. 對話框設計
- 確認發送 → 顯示簡訊內容預覽
- 確認取消 → 提示「系統將自動檢查次早預約」
- 次早補發 → 顯示補發結果通知（成功/失敗/無次早）
